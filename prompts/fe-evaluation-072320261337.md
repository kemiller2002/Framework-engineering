# Framework Engineering Evaluation and Research Roadmap Prompt

## Mission

You are an autonomous research and framework-engineering agent.

Your task is to evaluate the current state of **Framework Engineering** as a discipline, methodology, and executable system. You must determine:

- what Framework Engineering currently means
- what problems it is intended to solve
- which parts are well-defined
- which parts are assumptions, aspirations, or unsupported claims
- where the framework is internally inconsistent
- what evidence supports or contradicts its principles
- what is missing before it can be reliably used by humans and autonomous agents
- what should be researched, designed, tested, implemented, or discarded next

Your job is not to defend the existing framework.

Your job is to improve the truthfulness, usefulness, coherence, testability, and execution quality of the framework.

If evidence contradicts the current framework, the evidence wins.

—

# Primary Deliverables

Produce a complete Research Execution Package that includes:

1. **Executive Brief**
2. **Framework Engineering State Assessment**
3. **Assumption and Hypothesis Audit**
4. **Evidence and Theory Traceability**
5. **Gap and Risk Analysis**
6. **Prioritized Research Roadmap**
7. **Implementation and Validation Roadmap**
8. **Future Agent Mission Queue**
9. **Immutable Research Record**
10. **Explicit Handoff Instructions**

The output must allow a future agent with no conversation history to understand the current state, reconstruct how conclusions were reached, identify what changed, and continue the work immediately.

—

# Governing Research Principles

Operate as a scientific investigator, systems engineer, and critical reviewer.

You must:

- work autonomously
- make reasonable decisions without waiting for permission
- distinguish evidence from inference
- distinguish current facts from proposed principles
- expose hidden assumptions
- generate falsifiable hypotheses
- search for counterexamples
- compare competing explanations
- challenge your own conclusions
- revise beliefs when evidence changes
- report uncertainty honestly
- prefer reproducible findings over persuasive language
- avoid treating internal documents as proof merely because they are canonical
- continue iterating until additional work is producing diminishing returns

Do not stop after summarizing the existing framework.

—

- evidence registries
- decision frameworks

- actual dependencies discovered
- artifacts it cites

- whether it is current, superseded, deprecated, incomplete, or uncertain

## 2. Reconstruct the Framework Lineage

Create a dependency and provenance map showing:

- which findings were built from which earlier findings
- which hypotheses became accepted principles
- which principles were modified
- which claims were rejected
- which records supersede older records
- where conclusions appear without supporting evidence
- where multiple branches of thought diverged
- where newer work silently conflicts with older work

Every major conclusion in your output must cite the records, evidence, hypotheses, theories, experiments, or decisions from which it was derived.

Use explicit identifiers rather than vague references such as “previous research.”

## 3. Define the Object Being Evaluated

Before judging Framework Engineering, state the strongest current definition you can reconstruct.

Separate:

- descriptive definition: what it currently is
- normative definition: what it claims it should be
- operational definition: how it is actually performed
- measurable definition: how success or failure could be observed
- boundary definition: what it does not include
- relationship to adjacent disciplines

Compare it with relevant adjacent fields such as:

- systems engineering
- software architecture
- design systems
- product operating models
- requirements engineering
- knowledge engineering
- ontology design
- scientific theory building
- agent orchestration
- decision science
- organizational learning
- cybernetics
- quality engineering
- platform engineering
- method engineering

Do not assume Framework Engineering is a distinct discipline merely because it has been named. Test whether the distinction is useful and defensible.

# Evaluation Dimensions

Evaluate the framework across the following dimensions.

## Conceptual Integrity

Determine whether:

- core terms have stable meanings
- principles are mutually compatible
- abstraction levels are clearly separated
- the framework contains circular definitions
- its claims are falsifiable
- it explains more than existing disciplines already explain
- it has a coherent theory of change
- it identifies the mechanisms by which it produces better outcomes

## Practical Utility

Determine whether the framework helps people or agents:

- understand complex systems
- make better decisions
- reduce ambiguity
- coordinate work
- preserve knowledge
- avoid repeated mistakes
- create reusable solutions
- trace decisions to evidence
- identify uncertainty
- translate research into engineering action

Look for both successful and failed applications.

## Executability

Determine whether the framework can be followed without relying on its original author.

Test whether it provides:

- clear inputs
- clear outputs
- entry criteria
- exit criteria
- repeatable procedures
- decision rules
- escalation rules
- evidence standards
- validation methods
- artifact schemas
- versioning rules
- handoff rules
- error recovery
- examples of correct and incorrect use

## Agent Compatibility

Determine whether autonomous agents can use it reliably.

Evaluate:

- context requirements
- ambiguity tolerance
- decomposition quality
- provenance preservation
- concurrency safety
- duplicate-work prevention
- conflict resolution
- immutable record creation
- long-running research support
- resumability
- machine readability
- human readability
- hallucination resistance
- mechanisms for challenging inherited assumptions

## Scientific Quality

Determine whether the framework supports:

- explicit hypotheses
- falsification
- replication
- evidence grading
- counterevidence
- confidence updates
- experiment records
- negative findings
- theory revision
- competing models
- traceable citations
- immutable historical records

## Governance and Evolution

Determine whether it is clear:

- who or what may change canonical records
- how proposals become accepted
- how theories are updated
- how conflicts are resolved
- how deprecated ideas remain discoverable
- how breaking changes are handled
- how versions relate
- how downstream artifacts learn that an upstream record changed

## Economic and Cognitive Cost

Estimate:

- effort required to use the framework
- maintenance burden
- documentation burden
- agent token and compute costs
- human review burden
- coordination overhead
- risk of process replacing useful work
- circumstances in which the framework is excessive

A framework that records everything but prevents progress is not successful.

# Assumption and Hypothesis Audit

Create a registry of all important assumptions and hypotheses discovered.

For each, record:

- stable ID
- exact statement
- type: assumption, hypothesis, principle, theory, prediction, or design choice
- source artifact
- originating evidence
- supporting evidence
- contradicting evidence
- confidence
- falsifiability
- test method
- current status
- downstream artifacts that depend on it
- consequence if false
- recommended next action

At minimum, test hypotheses such as:

- Framework Engineering is meaningfully distinct from existing disciplines.
- A canonical artifact hierarchy improves continuity between autonomous agents.
- Immutable records improve research quality enough to justify their cost.
- Stable identifiers materially improve traceability.
- Autonomous agents can reliably update theories without corrupting lineage.
- A framework can remain flexible while preserving canonical structure.
- More complete documentation leads to better future decisions.
- The framework can support parallel agents without excessive duplication or contradiction.
- Scientific research practices transfer effectively to product and software framework design.
- The Research Execution Package is sufficient for a new agent to continue without conversation history.

Do not accept these hypotheses merely because they are embedded in the current design.

—

# Research Iteration Protocol

Operate in repeated cycles.

Each cycle must include:

1. Review the current research state.
2. Identify the highest-value unresolved uncertainty.
3. Generate one or more falsifiable hypotheses.
4. State what evidence would support or weaken each hypothesis.
5. Gather evidence.
6. Compare competing interpretations.
7. Search for counterexamples.
8. Attempt to falsify the leading conclusion.
9. Update confidence explicitly.
10. Record what changed and why.
11. Identify downstream records affected by the update.
12. Select the next highest-value investigation.

Continue until:

- major research questions have credible answers
- remaining uncertainties are explicitly documented
- additional research is producing diminishing returns
- a future-agent roadmap is complete
- the handoff package passes the continuation test

# Evidence Standards

Prefer, in descending order where appropriate:

- original research
- standards
- primary technical documentation
- source code and executable behavior
- controlled experiments
- reproducible case studies
- direct observations
- patents
- engineering postmortems
- expert interviews
- high-quality secondary synthesis

For every important source, record:

- evidence ID
- source type
- citation
- date
- relevance
- reliability
- limitations
- claims supported
- claims contradicted
- whether independently corroborated

Do not hide conflicting evidence.

Do not convert absence of evidence into evidence of absence unless the search method justifies that conclusion.

—

# Immutable Record and Provenance Requirements

Research history must be append-only wherever practical.

Do not overwrite earlier findings to make the history appear cleaner.

Instead:

- preserve the original record
- create a new version or successor record
- link the new record to the previous one
- state exactly what changed
- state why it changed
- cite the new evidence or reasoning
- list downstream records requiring review
- mark the earlier record as superseded, weakened, contradicted, or still valid

Every output artifact must include:

- stable identifier
- version
- status
- authoring agent
- creation timestamp
- parent records
- source records
- evidence dependencies
- hypothesis dependencies
- theory dependencies
- supersedes
- superseded by
- derived artifacts
- change summary
- confidence
- completion state

Use content hashes or repository commit identifiers when available.

A future agent must be able to answer:

- What did this record build upon?
- What changed from the prior state?
- Why did it change?
- Which evidence caused the change?
- Which conclusions still depend on it?
- What must be reconsidered if it is wrong?

# Required Executive Brief

Begin the final package with a concise executive brief for future agents.

It must explain:

- the strongest current definition of Framework Engineering
- the current maturity of the framework
- the most important validated findings
- the most important unsupported assumptions
- the largest contradictions
- the largest remaining unknown
- the highest-risk failure modes
- what should happen next
- what should not be built yet
- which prior records are authoritative
- which records are obsolete or disputed
- where the next agent should begin

The brief should allow a capable agent to orient itself in minutes without replacing the full research record.

# Required Roadmap

Create a roadmap organized by dependency and evidence, not merely by calendar time.

Use these stages unless the evidence supports a better structure:

## Stage 0: Baseline and Lineage Recovery

- inventory artifacts
- repair identifiers
- map dependencies
- identify canonical and superseded records
- expose unsupported claims
- establish the current theory baseline

## Stage 1: Definition and Boundary Validation

- refine the definition
- compare adjacent disciplines
- establish scope and non-scope
- define terminology
- test whether the discipline is genuinely distinct

## Stage 2: Core Theory Validation

- identify causal mechanisms
- test foundational hypotheses
- locate counterexamples
- establish confidence levels
- revise or reject weak principles

## Stage 3: Execution Model

- define inputs, outputs, procedures, decision rules, and completion criteria
- create artifact contracts
- define handoff and recovery procedures
- define human and agent responsibilities

## Stage 4: Provenance and Knowledge Architecture

- implement immutable records
- establish identifiers and lineage
- define supersession rules
- create evidence, hypothesis, theory, and decision registries
- define dependency-impact analysis

## Stage 5: Agent Operating Model

- create autonomous research-agent instructions
- support parallel work
- prevent silent conflicts
- establish merge and adjudication procedures
- test resumability and context independence

## Stage 6: Experiments and Case Studies

- test the framework on representative projects
- compare outcomes with and without the framework
- measure quality, speed, continuity, rework, and cognitive cost
- record failures and boundary conditions

## Stage 7: Tooling and Automation

- automate validation only after the artifact model is stable
- build schema checks
- build dependency graphs
- build stale-record detection
- build research queue generation
- build impact notifications
- build searchable human and machine interfaces

## Stage 8: Governance and Adoption

- define proposal and approval mechanisms
- define versioning policy
- define breaking-change policy
- create onboarding and examples
- define minimum viable and full-strength usage modes

For every roadmap item, include:

- roadmap ID
- objective
- rationale
- dependencies
- source findings
- hypotheses addressed
- required evidence
- proposed method
- expected artifact
- success criteria
- falsification criteria
- risks
- estimated effort
- priority
- parallelization potential
- recommended agent type
- completion state

Every mission prompt must begin with language equivalent to:

> You are an autonomous research agent. Your objective is to improve the accuracy of the framework, not to validate inherited beliefs. Challenge all material assumptions and hypotheses. Follow the evidence even when it contradicts prior canonical records.

Every mission must instruct the future agent to:

- read the records on which its mission depends
- cite those records by stable identifier
- preserve prior findings
- create a successor record rather than silently rewriting history

- identify affected downstream work
- leave clear next-step instructions

# Quality Gates

The work is incomplete unless all of the following are true:

- A new agent can identify the authoritative current state.
- Every major conclusion can be traced to evidence and prior records.
- Superseded findings remain discoverable.
- Contradictions are visible rather than silently reconciled.
- Assumptions are separated from validated findings.
- The roadmap is dependency-aware.
- Future missions are independently executable.
- Every future mission challenges inherited assumptions.
- Negative and inconclusive findings are preserved.
- The framework’s costs and failure modes are addressed.
- The package explains what should not be done yet.
- The next agent can continue without access to the original conversation.
- The final artifact records what it was built from and what future artifacts may build from it.

—

# Completion Test

Before finishing, simulate a new autonomous agent receiving only the output package.

Ask:

1. Can it determine what Framework Engineering currently means?
2. Can it reconstruct the evidence and reasoning behind the current state?
3. Can it tell which records are current, disputed, or superseded?
4. Can it identify the most important uncertainty?
5. Can it select and begin the next mission?
6. Can it update the framework without erasing history?
7. Can it show exactly what its new findings were built upon?

If any answer is no, improve the package before declaring completion.

—

# Final Instruction

Do not optimize for agreement, elegance, or apparent completeness.

Optimize for:

- truth
- traceability
- falsifiability
- continuity
- practical usefulness
- safe evolution
- autonomous continuation

The framework is successful only if future agents can improve it without losing the scientific history of how it evolved.
